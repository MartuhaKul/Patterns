# Патерни проектування на TypeScript

Лабораторна робота з вебпрограмування. Реалізовано **три класичні патерни проектування** GoF — по одному з кожної категорії.

## Зміст

| № | Категорія | Патерн | Файл |
|---|-----------|--------|------|
| 1 | Породжувальний (Creational) | **Singleton** (Одинак) | [src/singleton.ts](src/singleton.ts) |
| 2 | Структурний (Structural) | **Adapter** (Адаптер) | [src/adapter.ts](src/adapter.ts) |
| 3 | Поведінковий (Behavioral) | **Observer** (Спостерігач) | [src/observer.ts](src/observer.ts) |

## Структура проєкту

```
Patterns/
├── src/
│   ├── singleton.ts   # Породжувальний патерн
│   ├── adapter.ts     # Структурний патерн
│   ├── observer.ts    # Поведінковий патерн
│   └── index.ts       # Демонстрація всіх трьох
├── package.json
├── tsconfig.json
└── README.md
```

## Запуск

```bash
npm install         # встановити залежності (TypeScript, ts-node)
npm start           # запустити демонстрацію всіх патернів
```

Окремо кожен патерн:

```bash
npm run singleton
npm run adapter
npm run observer
```

Або зібрати у JavaScript:

```bash
npm run build
node dist/index.js
```

## 1. Singleton (породжувальний)

**Призначення.** Гарантує, що клас має лише **один екземпляр**, і надає до нього глобальну точку доступу.

**Коли застосовувати.** Коли в системі має існувати рівно один об'єкт певного типу: логер, конфігурація, пул з'єднань із базою даних, кеш.

**Як реалізовано.** Клас `Logger` має приватний конструктор (його не можна створити через `new` ззовні) і статичний метод `getInstance()`, який повертає єдиний екземпляр, створюючи його лише при першому виклику (lazy initialization).

```ts
const a = Logger.getInstance();
const b = Logger.getInstance();
console.log(a === b); // true — це той самий об'єкт
```

**Плюси:** контроль над єдиним екземпляром, відкладене створення, глобальний доступ.
**Мінуси:** ускладнює юніт-тестування (глобальний стан), порушує принцип єдиної відповідальності.

## 2. Adapter (структурний)

**Призначення.** Дозволяє об'єктам із **несумісними інтерфейсами** працювати разом, «перекладаючи» виклики одного інтерфейсу у виклики іншого.

**Коли застосовувати.** Коли потрібно інтегрувати сторонню бібліотеку, чий API не збігається з очікуваним інтерфейсом вашого коду, і змінювати ні те, ні інше — небажано.

**Як реалізовано.** Застосунок працює з єдиним інтерфейсом `PaymentProcessor.pay(amountUah)`. Існують дві платіжні системи з різними API: `PayPalApi.sendPayment(usd)` і `StripeApi.makeCharge(cents)`. Класи `PayPalAdapter` та `StripeAdapter` приховують різницю — конвертують гривні в потрібну валюту та делегують виклик відповідній бібліотеці.

```ts
const processor: PaymentProcessor = new StripeAdapter(new StripeApi());
processor.pay(1000); // клієнтський код не знає про Stripe API
```

**Плюси:** відокремлення клієнта від конкретних реалізацій, легка заміна постачальника.
**Мінуси:** збільшує кількість класів.

## 3. Observer (поведінковий)

**Призначення.** Визначає залежність «**один до багатьох**»: коли видавець (Subject) змінює стан, усі підписники (Observer) автоматично отримують сповіщення.

**Коли застосовувати.** Системи подій, реактивний UI, push-сповіщення, RSS-стрічки, чат-кімнати — будь-де, де одна подія має оновити кілька незалежних компонентів.

**Як реалізовано.** Клас `NewsChannel` тримає список підписників (`Observer[]`) та надає методи `subscribe`, `unsubscribe`, `publish`. Підписники `EmailSubscriber` і `SmsSubscriber` реалізують спільний інтерфейс `Observer<string>` із методом `update(news)` — кожен реагує на новину по-своєму.

```ts
const channel = new NewsChannel();
channel.subscribe(new EmailSubscriber('alice@example.com'));
channel.subscribe(new SmsSubscriber('+380501234567'));
channel.publish('Нова стаття опублікована'); // сповіщені обоє
```

**Плюси:** слабка зв'язаність видавця та підписників, динамічне додавання/видалення слухачів.
**Мінуси:** порядок сповіщень не визначений, можливі витоки пам'яті, якщо забути відписатися.

---

## Висновок

Усі три патерни розв'язують різні задачі:
- **Singleton** керує **створенням** об'єкта (один на всю програму).
- **Adapter** змінює **структуру** взаємодії з чужим кодом (приводить інтерфейс до потрібного).
- **Observer** організовує **поведінку** — реакцію набору об'єктів на події одного.

Разом вони демонструють три фундаментальні категорії GoF-патернів і типові проблеми, які вирішує об'єктно-орієнтоване проектування.
